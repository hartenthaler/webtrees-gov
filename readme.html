Description

back to top

This module will display in a new tab information on the places mentioned in the "facts and events" tab of a person based on information of the GOV system (Genealogisches Ortsverzeichnis).

Requirements

back to top

This module requires webtrees version 1.5.0 or later. It was tested with webtrees version 1.5.1.

Installing the module

back to top

Installation is simple:

  • upload the folder 'gov' (don't use capital letters) to your server's webtrees folder /modules_v3
  • login to webtrees as administrator and go to Module Administration
  • look down the list of installed modules for "GOV" and ensure it is ticked as "enabled"
  • in the "Modules" submenue "Tabs" you can reorder the tabs and you can define who can see the GOV tab.

Enter data

back to top

These are the instructions on getting your GOV interface to work.

The additional GOV information will only be shown if at least one fact has a place with a GOV id attached to it. Attaching a GOV id can be done by an administrator through the generic place-location interface (located at the GOV preferences page) or by specifying a "_GOV" record or a "NOTE" record containing a GOV id with an event, even though this is not according to the GEDCOM standard. Other genealogy applications may not recognise a "_GOV" tag.

Adding GOV ids one by one to your GEDCOM file (not recommended)

This method is included so that you may have an idea of how a GEDCOM file stores GOV data. You do not need the GOV module to use this procedure. The GOV ids for an event can only be added directly to a GEDCOM file. There are two possibilities do this for a PLAC record:

2 PLAC <Placename>
3 _GOV <GOV id>

or

2 PLAC <Placename>
3 NOTE GOV: <GOV id>

For example:

1 BIRT
2 DATE 24 JUL 1957
2 PLAC Sigmaringen, Sigmaringen, Baden-Württemberg, DEU
3 NOTE GOV: SIGGENJN48OC
1 EDUC Abitur
2 DATE 19 MAY 1976
2 PLAC Freiburg, Freiburg, Baden-Württemberg, DEU
3 _GOV FREURGJN38WA

If you like to use the GEDCOM user defined tag _GOV, you may add this tag in the administrator menue "Family Trees", tab "Edit options" under "Advanced fact settings" in the field "Advanced place name facts".

Add a place with a GOV id using webtrees GOV module (recommended method)

The GOV module is designed to work with locations in tree-like fashion. If we were to consider a suburb of London, the tree would take the order 'England, London, Hackney'. This is not how we enter place names in our family data, but it is how we manage it in the Google Map module or in the GOV module. This will give access to useful place lookup features as we build up structured information. It will help you find groups of people from the same locale. And it conforms to the GEDCOM specification, a feature of webtrees.

And, before you start with any GOV endeavours, review your data. Make sure for all place names that

  • spelling is consistent
  • there are no 'almost' duplicates
  • places are in their right country
  • places you describe are in true tree fashion (hierarchical)
  • they have the spelling which is used today, so avoid for example historical place names like "Prussia"
    • instead of "Neumittelwalde, Groß Wartenberg (Landkreis), Niederschlesien, Preußen"
    • use better "Międzybórz, Oleśnica, Dolnośląskie, POL" for this place

Only then will the module make it easy for you to connect family data with GOV.

This module uses a new database table to store place text and GOV information. Existing places can be imported and location information can be added using the GOV webservice. The location information is held outside the GEDCOM (and can be shared between family trees in webtrees) and location data is entered only once for each place.

Backup of location data is available by export of each new place database table to a text file (separated with ";"). Bulk additions can be performed by text file import (with reservations for specific place structure and spelling).

Using the tree structure mentioned above, we start to build location data from the top down – which, in 99% of cases, will mean starting by entering a country. When that is done, we move to the next layer (state, county, whatever is appropriate for your locale.)

Custom translations

back to top

All language items are included in the "WT_I18N::translate()", i.e. "gettext", format. This means you can translate these into any other language using webtrees "custom translation" feature, described here: Wiki: Custom translation. Instead of adding your custom translation file(s) to the folder data/language, you have to keep them in the folder modules_v3/gov/language.

You can use the _default.pot file added in the module language folder, to start translate the module in other languages. The _default.pot file contains all strings needed to be translated. You can open this file in any texteditor. Save your file as <language ISO 639-1 code>.po file. To use the translated *.po file on the module you need to convert this file to a *.mo file. This can be done with a programm like Poedit. But it is not mandatory. Just send me the translated *.po file and I will turn it in to a *.mo file and add it to the module package.

The German translation file is already added to the module language folder.

Design Decisions

back to top

This design of this module follows the existing Google Maps module to make it easy for an webtrees administrator to use it.

This module doesn't introduce a new place management system for webtrees. There are some ideas to manage places as level 0 records in the GEDCOM file, but this is up to now not supported by webtrees.

For each place only the GOV id is stored in a database table. Whenever actual data for a place has to be shown in the new GOV tab, this information comes from the GOV database using the GOV webservice. So the amount of additional data which is stored inside the webtrees database is minimal and the data itself is always actual. The disadavantage is that it needs some time to fetch the actual data.

The new GOV functions are implemented as a webtrees module, so these functions are only available inside the new GOV tab, but nowhere else inside webtrees. There are no plans to modify other webtree modules, reports or functions to make the GOV functions available there.
